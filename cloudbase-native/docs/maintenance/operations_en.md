# Operations

## Local Development

1. Open `cloudbase-native` in WeChat DevTools.
2. Configure the real Mini Program AppID.
3. Enable Cloud Development in the selected environment.
4. Install dependencies inside each Cloud Function directory if needed.
5. Upload Cloud Functions from DevTools.

## Launch Window Checklist

Use this when CloudBase is finally opened so the paid window stays short.

1. Create collections: `users`, `tasks`, `task_archives`.
2. Add the recommended indexes before importing or using real data.
3. Upload `taskGateway`.
4. Upload `generateTaskByAI`.
5. Upload `moderateContent`.
6. Upload `subscribeScheduler`.
7. Fill `miniprogram/config/subscribe.js` with the Mini Program subscribe template IDs.
8. Set environment variables only after the function upload succeeds.
9. Create the timed trigger for `subscribeScheduler`.
10. Run a smoke test in this order:
   - bootstrap profile
   - request subscribe authorization
   - create normal task
   - create self-assigned task
   - accept task
   - verify creator-side subscribe notification on accept
   - update progress
   - submit review
   - verify creator-side subscribe notification on review submit
   - continue review
   - complete task
   - delete archive
   - rework task
   - accept rework
   - reject rework
   - cancel rework
   - refresh overdue task schedule
   - run `subscribeScheduler` manually once
11. Only after the smoke test passes, switch test data to real usage.

## Recommended Indexes

### `tasks`

- `creatorId + updatedAt`
- `assigneeId + updatedAt`
- `seedKey`
- `status + dueAt`

### `task_archives`

- `ownerId + updatedAt`
- `sourceTaskId`

## Maintenance Notes

- The current implementation uses pull-based refresh, so there is no socket
  service to maintain.
- Normal production functions should use:
  - `taskGateway`: `256MB / 20s`
  - `generateTaskByAI`: `256MB / 20s`
  - `moderateContent`: `256MB / 20s`
  - `subscribeScheduler`: `256MB / 20s`
- Subscribe messages are one-shot. After a successful send, the user scene is
  marked as `pending_reauth` until the user grants permission again.
- The frontend sends the last known `updatedAt` value for critical task writes.
  If the task changed on the server, the write is rejected and the function
  returns a fresh full dashboard snapshot.
- Challenge tasks are generated by deterministic daily seeds.
- The dashboard intentionally returns the full archive list because the single
  shell archive tab relies on it directly.
- Archive cleanup and long-term retention policy should be added later with a
  scheduled function.
- Basic sensitive-word checks are implemented locally in `taskGateway`, and
  `moderateContent` can add model-based moderation when configured.
