# 技术选型

## 最终采用

- 微信原生小程序作为客户端
- CloudBase 云函数作为后端执行边界
- CloudBase 数据库作为持久层
- CommonJS JavaScript 作为当前主要代码形式

## 为什么这样选

### 原生小程序

选择原生而不是 Taro/React，主要因为：

- 目标平台就是微信小程序
- 少一层框架封装，调试更直接
- 页面和交互性能更可控

### Cloud Functions

选择云函数而不是独立 Express 服务，主要因为：

- 部署更轻
- 更符合微信生态
- AI 和订阅消息密钥都能留在服务端
- 任务规则仍能统一维护

### Cloud Database

选择 CloudBase 数据库而不是 MongoDB，主要因为：

- 减少额外基础设施
- 和当前运行时天然配套
- 当前任务模型适合文档存储

### 拉取刷新 + 冲突保护

选择全量刷新和冲突保护，而不是 WebSocket 推送，主要因为：

- 当前产品不需要高频协同编辑
- 维护成本更低
- 写后刷新已经足够满足一致性要求

## 当前明确不使用

- Taro
- React
- Express
- MongoDB
- WebSocket 同步
